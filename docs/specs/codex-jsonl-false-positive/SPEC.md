# Spec: Codex JSONL False-Positive Prevention — `ClassifyFromEvents`

**Status:** Draft
**Supersedes:** `docs/specs/codex-jsonl-false-positive.md`
**Effort:** Small (~60 LOC + tests)
**Research:** [research/codebase.md](research/codebase.md)

## Problem

`pkg/errors.Classify()` scans a raw string for error patterns (auth failures, rate limits, etc.). When callers pass concatenated Codex JSONL output — which embeds MCP tool results, command output, and API responses inside structured fields — the embedded content triggers false positives. For example:

- An MCP tool result containing `"authorization": null, "error": null` matches `auth_error`
- A command execution output containing `API Error: 500 in server.log` matches `upstream_api_error`
- A file diff showing `permission denied` in a test fixture matches `permission_denied`

The sidecar's internal `trackEventMetrics` already filters by event type (only `agent_message`), but this protection is not available to external callers (daemon log replay, SDK consumers, test tooling). Additionally, `trackEventMetrics` does not scan `error`-type events, missing legitimate backend errors.

## Solution

Add `ClassifyFromEvents` to `pkg/errors/classify.go` — a function that accepts structured events and only scans text from `agent_message` and `error` types, ignoring `tool_use`, `tool_result`, `system`, and all other event types. This formalizes the sidecar's existing event-type filtering as a reusable public API.

## Design

### Event Input Type

```go
// EventForClassification is a minimal event representation for error scanning.
// Callers extract Type and Text from their native event structures.
type EventForClassification struct {
    Type string // "agent_message", "error", "tool_result", etc.
    Text string // The text content to scan (only used for qualifying types)
}
```

This avoids importing sidecar-specific types (`Event`, `NormalizedAgentMessage`) into `pkg/errors`. Callers map their event representations to this struct.

### Core Function

```go
// ClassifyFromEvents scans only agent-authored text and error messages
// for known error patterns. Tool results, system events, and other
// event types are deliberately skipped to avoid false positives from
// embedded content (MCP results, command output, file diffs).
func ClassifyFromEvents(events []EventForClassification) ErrorCategory {
    var buf strings.Builder
    for _, evt := range events {
        switch evt.Type {
        case "agent_message", "error":
            if evt.Text != "" {
                buf.WriteString(evt.Text)
                buf.WriteByte('\n')
            }
        }
    }
    if buf.Len() == 0 {
        return CategoryNone
    }
    return Classify(buf.String())
}
```

Key decisions:
- **Delegates to `Classify()`** — single source of truth for regex patterns
- **Scans `error` events** — these carry backend-level errors (`API Error: 503`, auth failures) that the current `trackEventMetrics` misses
- **No `tokenCount` parameter** — startup crash detection uses `DetectStartupCrash()` separately with int args; mixing concerns would complicate the API
- **No Codex-detection heuristic** — callers that have structured events already know their backend type; `isCodexOutput()` from the old spec is unnecessary

### Event Types Scanned vs. Skipped

| Event Type | Scanned | Rationale |
|---|---|---|
| `agent_message` | Yes | Agent's own text — the primary signal for error classification |
| `error` | Yes | Backend error messages (API errors, auth failures) |
| `tool_use` | No | Tool invocation metadata, no error-relevant text |
| `tool_result` | No | **Primary false-positive source** — contains command output, MCP results, file content |
| `system` | No | Stderr lines, hooks, thread lifecycle — noisy, not error-classifiable |
| `result` | No | Turn completion metadata with usage stats, not text |
| `progress` | No | Status updates, not error text |
| `exit` | No | Generated by the sidecar itself, already has classification |

## Integration

### Sidecar (`cmd/sidecar/ws.go`)

Two options, in order of preference:

**Option A (minimal change):** Also accumulate `error` event text in `trackEventMetrics`:

```go
case "error":
    if msg, ok := data["message"].(string); ok {
        s.agentTextBuf.WriteString(msg)
        s.agentTextBuf.WriteByte('\n')
    }
```

This closes the gap where backend error events are missed, without changing `classifySessionError`'s call to `Classify()`. The sidecar already has event-type filtering via the `switch` statement.

**Option B (full switch):** Replace `agentTextBuf` accumulation with an `[]EventForClassification` slice, and call `ClassifyFromEvents` in `classifySessionError`. More correct but higher churn for the same result in the sidecar context.

Recommendation: **Option A** for the sidecar, expose `ClassifyFromEvents` for external callers.

### Daemon Log Replay (`pkg/api/`)

When the daemon parses NDJSON session logs for post-hoc error classification (e.g., session detail API, retry decisions), it should:

1. Parse each NDJSON line into `{type, data}`
2. Extract text: `data.text` for `agent_message`, `data.message` for `error`
3. Build `[]EventForClassification`
4. Call `ClassifyFromEvents(events)`

This replaces any future temptation to call `Classify()` on concatenated raw log lines.

## Testing

### Unit Tests (`pkg/errors/classify_test.go`)

```go
func TestClassifyFromEvents(t *testing.T) {
    tests := []struct {
        name   string
        events []EventForClassification
        expect ErrorCategory
    }{
        {
            name: "agent reports auth error",
            events: []EventForClassification{
                {Type: "agent_message", Text: "Authentication failed: invalid API key"},
            },
            expect: CategoryAuthError,
        },
        {
            name: "tool result with auth-like content ignored",
            events: []EventForClassification{
                {Type: "tool_result", Text: `{"authorization": null, "error": null}`},
                {Type: "agent_message", Text: "The API returned a successful response."},
            },
            expect: CategoryNone,
        },
        {
            name: "error event with upstream API error",
            events: []EventForClassification{
                {Type: "error", Text: "API Error: 503 Service Unavailable"},
            },
            expect: CategoryUpstreamAPI,
        },
        {
            name: "command output with permission denied ignored",
            events: []EventForClassification{
                {Type: "tool_result", Text: "ls: /root: permission denied"},
                {Type: "agent_message", Text: "I don't have access to that directory. Let me try another approach."},
            },
            expect: CategoryNone,
        },
        {
            name: "mixed events only scans qualifying types",
            events: []EventForClassification{
                {Type: "system", Text: "stderr: rate limit exceeded"},
                {Type: "tool_result", Text: "Error: authentication error in test fixture"},
                {Type: "agent_message", Text: "All tests passed."},
            },
            expect: CategoryNone,
        },
        {
            name: "empty events",
            events: nil,
            expect: CategoryNone,
        },
        {
            name: "MCP result with error-like JSON ignored",
            events: []EventForClassification{
                {Type: "tool_result", Text: `"type":"error","Internal server error"`},
                {Type: "agent_message", Text: "The MCP tool returned data successfully."},
            },
            expect: CategoryNone,
        },
    }
    // ...
}
```

### Regression Tests

These cases cover the specific false-positive triggers from production:

1. **Codex MCP JSON with `authorization`/`error` fields** in `tool_result` -> `CategoryNone`
2. **Codex command output with `API Error: 500`** in `tool_result` -> `CategoryNone`
3. **Same strings in `agent_message`** -> correctly classified (not a false positive)
4. **Same strings in `error` event** -> correctly classified

## Migration

The existing `Classify()` function remains unchanged and public. `ClassifyFromEvents` is additive. No breaking changes.

The old spec at `docs/specs/codex-jsonl-false-positive.md` proposed a `NormalizedEvent` type and `isCodexOutput()` detection function that are no longer needed given the current architecture. This spec supersedes that proposal.

## Out of Scope

- Changing `Classify()` signature or regex patterns
- Agent-quoting-tool-output false positives (the agent's own text saying "I got an authentication error" is a correct classification — the agent encountered an auth error)
- `DetectStartupCrash` integration (orthogonal concern, already handled separately)
- Daemon-side session monitoring (separate feature)
